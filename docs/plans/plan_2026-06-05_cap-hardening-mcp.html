目标

Goal
在 team-context-mcp（三仓中间的薄门面层）为 6 大能力补强 + 设计检查点新增/复用 ≤8 个 MCP 工具，每个都是对 tc-multica 对应端点的薄封装——业务逻辑（度量/去重/根因）一律留 tc-multica、绝不下沉本仓。波次重排后按：波次1 design_review（零依赖先 ship）+ delivery_metrics（端点 ready 后）→ 波次3 去重(对接 pg_bigm)/打标/拆任务 → 波次4 日志/根因 → 波次5 组件目录。

完成标准

  • 每个新工具在对应 server 的 tools/list 出现且 inputSchema.type==='object'（过 input-schema.test.ts，顶层不得 emit oneOf）
  • 每个新工具有 handler 单测，用 mock 打桩 tc-multica 端点、不真连后端；CI build-test 全绿
  • 薄门面纪律=结构性判据：每个新 handler 内无 sort/截断/算分/embedding 调用，仅『校验入参→MulticaClient.req(/api/...)→透传』；需访问 tc-multica 数据的方法都落 packages/shared/src/multica-client.ts（行数仅软参考，非判据）
  • design_review 走 remote、不落本地 HTML：复用 plan_request_review 的 label+comment，『设计-已选型』终态用 remote 的 MulticaClient.addLabel/removeLabel/updateIssue 重写（不复用 local 的 plan_approve），能在设计 issue 上跑通三态机
  • delivery_metrics 启动前 tc-multica 端点须可 curl 通（非仅书面锁 schema）；端点未就绪则只写 mock 占位不合并空壳；README 工具表 + local/remote/total 计数同步（当前 23）

方案

## 仓库角色 team-context-mcp 是三仓【中间薄门面层】——把 tc-multica 长出的端点包成 agent 与 autopilot 能调的 MCP 工具，自身不实现业务逻辑。现有 23 工具 = local 13 + remote 10。 ## v1.1 变更摘要（评审 3 blocker） - design_review transport 纠正：plan_request_review 在 remote、plan_approve 在 **local 且做本地文件 I/O**，跨 transport 无法单工具复用 → 定走 remote 不落本地 HTML。 - 波次重排：零依赖的 design_review 提到 delivery_metrics 前，给夹心层不空转的起点。 - 薄门面判据改结构性；松绑飞书措辞；delivery_metrics 加端点可 curl 通 gate；dedup 改对接 pg_bigm。 ## 薄门面第一纪律（判据=结构性，非行数） 度量/去重/根因/打标逻辑留 tc-multica，本仓只透传。判据：**handler 内无 sort/截断/算分/embedding 调用**（行数是软参考——should_i_use_ai/betting_table_capture 也在 remote 且超 60 行，remote≠thin）。划分：聚合多源/操作 issue/需飞书→remote；只读本地代码产物→local。注意『remote』是常驻/transport 概念，TCM-1/3/4/5/6/7 只需 MulticaClient、不需 FeishuClient，落 remote 的真实理由是『常驻独占 token + 面向全工作区』。 ##
